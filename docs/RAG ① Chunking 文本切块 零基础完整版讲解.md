## 📌 前置零基础前提

> 学习 Chunking 之前，先理解两个背景：

- **RAG 需要把文档转向量存入向量库**
- **向量模型不能直接接收长篇大段文章**，单次只能处理一小段文本

所以必须做 **Chunking 切块**——把长文档切割成向量模型能吃的「小口食物」。

---

## 一、什么是 Chunking（文本切块）

> **定义**：Chunking 中文叫「文本切块 / 分块」，是 RAG 离线建库的第一步；把长篇、完整的原始大文档，切割成长度适中、语义独立、上下文完整的短小文本片段（片段就叫 **Chunk 块**）。

**通俗大白话**：把一本厚书，**裁成一张张不打碎完整句子、不打断一段话意思的小纸条**。

---

## 二、为什么 RAG 必须做切块？（底层原理）

三个核心底层原因，零基础必懂：

| # | 原因 | 说明 |
|---|------|------|
| 1 | **Embedding 向量模型有最大长度限制** | 向量化模型一次读不了整篇论文、长篇文档，超长文本直接报错 / 计算失效；只能输入短文本 |
| 2 | **检索精准度更高** | 整篇文档向量化，一大段混杂无关信息；切成小片段后，用户提问只匹配最相关的一小段内容，不会混入冗余垃圾信息 |
| 3 | **保护语义完整性** | 控制片段大小，既不切碎一句话逻辑，又不会因为片段太长稀释核心语义 |

---

## 三、核心专业原则（最重要）

> **Chunking 不是随便切断文字！！不能按字数生硬截断句子。**

✅ **核心标准**：优先保证切块后，**每一个 Chunk 语义完整、逻辑独立**。

简单说：**不能把一句话、一个完整知识点从中间劈断**。

## 五、关键补充：重叠切片（Overlap）原理

> 实际 RAG 工程**不会切得干干净净**；会设置 **20% 左右文本重叠**。

### 5.1 原理

很多知识点跨两个段落边界；如果切得干干净净，边界知识点会被切碎、检索不到。**相邻两个块保留一小段重复文字，保住跨片段语义**。

### 5.2 举例

> Chunk 2 尾部，复制 20 个字拼接到 Chunk 3 开头，两条规则边界信息不丢失，不影响检索精度。

## 九、Chunking 文本切块：到底用啥切？（零基础全解）

> 整体分为两大类：**按符号 / 算法自动切**（工程最常用）、**大模型智能切**；日常 RAG 开发不用手动剪切，全部靠代码工具自动完成。

### 9.1 第一类：底层切割依据（靠什么标记切断文字）

程序按照文本自带分隔符号切割，**优先级：结构符号 > 标点符号 > 纯字数**。

#### ① 结构分隔符（首选！语义最完整）

- **依据**：文档自带格式切割：标题、换行符、列表符号、章节标签
- **举例**：`#` 一级标题、`##` 二级标题、`1./2.` 有序列表、`-` 无序列表、段落换行
- **适用**：Markdown、Word、带目录的规范文档

#### ② 标点分隔符

- **依据**：按照语义终止标点切断：句号 `。`、分号 `；`、换行符 `\n`
- **原则**：只在完整句子末尾切断，不切句子中间

#### ③ 固定字数（最后备选，最差）

- **依据**：不看语义，单纯按 200 / 500 汉字硬性截断
- **适用**：仅用于无格式杂乱纯文本

### 9.2 第二类：主流切割算法（程序怎么切）

RAG 行业 **4 大标准切块算法**，**从上到下优先选用**：

| 优先级 | 算法 | 原理 | 优势 / 适用场景 |
|--------|------|------|-----------------|
| 🥇 1 | **递归字符切块**（最常用 ✅） | 从大分隔符 → 小分隔符层层往下切；先切标题段落 → 太长再切句子 → 最后才按字数切 | 最大程度保住语义完整性，适配 90% 普通文档 |
| 🥈 2 | **结构化标题切块** | 只按照文档层级标题拆分，同一个小节内容合成一个 Chunk | 教材、手册、规章制度、长报告（上面报销文档首选这个） |
| 🥉 3 | **语义相似度切块**（高精度） | 调用 Embedding 向量模型，把语义不相关的句子切开；语义相近的句子保留在同一个 Chunk | 切块逻辑最贴合人类理解；缺点：算力慢、消耗资源高 |
| 4 | **固定大小滑动切块**（最简单） | 纯按固定字数 + 滑动重叠窗口切割，自带 10%~20% 重叠；不看语义 | 容易切碎知识点，只适合杂乱无格式网页文本 |

### 9.3 第三类：实操工具（我们写代码用什么切）

普通人 / 开发者**直接调用现成工具**，不用自己写切割算法：

- **主流框架工具**（开发必用）
  - LangChain 文本分割器
  - LlamaIndex 切块工具
  - *封装好上面全部算法，一行代码开启重叠切块*

- **平台自带工具**（零代码）
  - Chroma、Milvus 向量库自带一键切块
  - RAG 平台内置分块功能，不用写代码

- **大模型智能切块**（高阶）
  - 让 LLM 判断知识点边界，自主切割 Chunk
  - 适合法律、医疗专业长文本

### 9.4 通俗总结（直接背）

> **切块优先用递归算法 + 文档标题 / 标点符号；开发用 LangChain 工具自动切割；尽量不用纯字数硬切；复杂专业文本用向量语义切块；所有工程切块默认开启 10%~20% 文本重叠。**

## 十、Embedding 向量化（衔接 Chunking 的下一步）

> **衔接上文流程**：Chunking 切好文本片段 → **Embedding 向量化**（本节）→ 存入向量数据库。

### 10.1 零基础通俗定义

> **Embedding（向量化）**：把人类能看懂的文字、句子、文本块，**翻译成计算机能看懂的一串超长数字（高维数组 / 向量）**。

- 笔记原文释义：向量化 = 文本转高维数字向量（**只能由 Encoder 类模型完成**）
- ✅ 大白话类比：**给每一段话发一张专属的「数字身份证」**
- 人类看：文字语义、句子意思
- 计算机看：只有一堆小数数字；计算机只会对比数字、不会读懂文字

### 10.2 什么是高维向量？（最简单例子）

> **通俗理解**：维度 = 这组身份证有多少个数字字段。

- **低维（举例）**：短句「下雨天带伞」→ 一组 5 个数字：`[0.21, 0.78, 0.13, 0.66, 0.35]`
- **高维（工程真实）**：一段报销 Chunk 文本 → **1536 个小数**组成的长串数字，这就是高维向量

这一整串超长数字，就是文本对应的「**高维数字向量**」。

### 10.3 核心重点：为什么只能 Encoder 模型做 Embedding？（必背考点）

回顾 Transformer 两大模块：

| 模块 | 特性 | 是否能做 Embedding |
|------|------|-------------------|
| **Encoder 编码器** | 双向阅读（同时看前后文），擅长理解语义、提取文本特征 | ✅ **唯一能做** |
| **Decoder 解码器**（GPT 类模型） | 单向续写，擅长聊天生成文字 | ❌ 不能做 Embedding |

> **底层逻辑**：向量化需要**完整读懂整段文字的语义**；只有双向注意力的 Encoder 能精准提取语义特征，转换成对应数字向量。

### 10.4 最关键特性（RAG 能运行的根本）

> **语义相近的文本，生成的高维向量数字距离很近；语义无关的文本，向量距离很远。**

计算机不靠读文字，**靠对比两组数字距离**判断两段话像不像！

### 10.5 直观实操案例（零基础彻底看懂）

把 3 句短句送入同一个 Encoder Embedding 模型做向量化：

| 序号 | 文本 | 与 A 的关系 |
|------|------|-------------|
| **A** | 北京今天下雨，出门需要带雨伞 | —— |
| **B** | 今日京城小雨，外出建议携带雨具 | 语义几乎和 A 一样 |
| **C** | 公司报销发票需要提交 PDF 格式 | 完全不相关 |

向量化结果（直观示意）：

- A 向量 和 B 向量：**数字整体高度接近、空间距离很近** ✅
- A 向量 和 C 向量：**数字差异极大、空间距离非常远** ❌

> **RAG 检索原理**：用户提问也转成向量，数据库匹配距离最近的 Top-K 向量文本，就是最相关资料。

### 10.6 串联整条 RAG 链路（衔接前面全部知识点）

```
1. 原始长文档
   │
   ▼
2. Chunking 切块（切成语义完整小文本）  ← 第七章
   │
   ▼
3. 【Encoder 模型执行 Embedding 向量化】文本转高维向量  ← 本节
   │
   ▼
4. 向量 + 原文存入向量数据库
   │
   ▼
5. 用户提问向量化检索（相似度匹配 Top-K）
   │
   ▼
6. LLM（GPT-Decoder）基于检索结果生成答案
```

### 10.7 极简背诵总结（直接背）

> **Embedding 向量化**：由 Encoder 编码器模型执行；将人类文本转为计算机可识别的高维数字向量；**语义相似文本向量距离更近**；是 RAG 检索的核心步骤，搭建文字和计算机数据的桥梁。

## 十一、向量相似度 / 距离算法详解（Embedding 的配套知识）

> 既然 Embedding 把文本变成了数字向量，**怎么判断两个向量像不像？** 这就要用到**相似度 / 距离算法**——它是向量数据库检索的「打分器」。

### 11.1 核心概念定义

#### 11.1.1 Embedding 嵌入向量

将文本、图像、音频、表格等**非结构化数据**，通过预训练模型映射为**固定长度、低维稠密的浮点向量**。

- **核心特性**：语义相近的数据，在向量高维空间中空间位置更接近；语义无关向量空间距离更远
- **常见模型**：BGE、text-embedding、E5、MiniLM、CLIP（多模态）

#### 11.1.2 向量相似度

量化两个 Embedding 向量语义接近程度的数值指标，是**语义检索、匹配、去重**的核心依据。

- **相似度值**：数值越大，向量越匹配
- **距离值**：数值越小，向量越匹配

### 11.2 主流相似度 / 距离算法详解

> 设向量 `a = [a₁, a₂, ..., aₙ]`、`b = [b₁, b₂, ..., bₙ]`，向量维度为 `n`。

#### 11.2.1 余弦相似度 Cosine Similarity（NLP 首选）

**计算公式**：

```
cos(a, b) = Σ(aᵢ × bᵢ) / ( √Σaᵢ² × √Σbᵢ² )
```

**值域与含义**：值域 `[-1, 1]`

| 值 | 含义 |
|----|------|
| **1** | 向量方向完全一致，语义高度相同 |
| **0** | 向量正交，语义无关联 |
| **-1** | 向量方向完全相反，语义对立 |

**优势**：仅衡量向量方向，**不受向量模长（长度）影响**，适配文本 Embedding；是 RAG、语义匹配的标准算法。

**衍生：余弦距离**：`cos_dist = 1 - cos_similarity`，值域 `[0, 2]`，向量数据库排序通用距离。

#### 11.2.2 点积 Dot Product

**公式**：

```
dot(a, b) = Σ(aᵢ × bᵢ)
```

> **关键特性**：向量完成 **L2 归一化**后，点积结果**完全等价于余弦相似度**。
>
> **工程价值**：计算量极小，FAISS、Milvus、Chroma 等向量库底层全部采用「**归一化 + 点积**」加速检索。

#### 11.2.3 欧氏距离 L2 Distance

**公式**：

```
d(a, b) = √Σ(aᵢ - bᵢ)²
```

- **数值越小**相似度越高
- **缺陷**：受向量模长干扰严重，文本 Embedding 场景极少使用，多用于图像像素匹配

#### 11.2.4 曼哈顿距离 L1 Distance

**公式**：

```
d(a, b) = Σ|aᵢ - bᵢ|
```

- 对异常值敏感，几乎不用于文本向量检索

### 11.3 算法对比与选型标准

| 算法 | 适用场景 | 优点 | 缺点 | 推荐度 |
|------|----------|------|------|--------|
| **余弦相似度** | 文本 RAG、语义搜索、问答匹配 | 不受向量长度影响，语义匹配精准 | 原生计算开销略高于点积 | ⭐⭐⭐⭐⭐ |
| **归一化点积** | 向量数据库大规模检索 | 计算速度最快，硬件友好 | 必须提前归一化向量 | ⭐⭐⭐⭐⭐ |
| **欧氏距离** | 图像特征、数值特征匹配 | 直观空间距离 | 模长干扰大，文本效果差 | ⭐⭐☆☆☆ |
| **曼哈顿距离** | 低维离散数据 | 计算简单 | 高维 Embedding 效果差 | ⭐☆☆☆☆ |

**选型规范**：

- 离线业务计算相似度 → **直接使用余弦相似度**
- 向量库在线检索 → **统一对向量 L2 归一化，使用点积替代余弦相似度提速**

### 11.4 完整业务标准流程

#### 11.4.1 知识库入库流程（离线）

```
① 文档切片、清洗、分段（Chunking）
   │
   ▼
② 调用 Embedding 模型生成文本向量
   │
   ▼
③ 向量执行 L2 归一化
   │
   ▼
④ 向量 + 文档元数据存入向量数据库
   │
   ▼
⑤ 建立索引（IVF_FLAT、HNSW 等 ANN 索引）
```

#### 11.4.2 用户查询召回流程（在线）

```
① 用户输入 Query 清洗预处理
   │
   ▼
② Query 生成 Embedding 向量并归一化
   │
   ▼
③ 向量库使用点积（等价余弦）检索 Top-K 相似向量
   │
   ▼
④ 根据相似度阈值过滤低相关文档
   │
   ▼
⑤ 重排序（Rerank）提升匹配精度
   │
   ▼
⑥ 输出匹配文档给大模型生成回答
```

## 十二、全文总结：一张图贯穿整条 RAG 链路

```
                原始长篇文档
                      │ 【Chunking 文本切块】
                      ▼
              语义完整的小 Chunk 块
                      │ 【Embedding 向量化 · Encoder 模型】
                      ▼
             高维数字向量（+ 原文本）
                      │ 【存入向量数据库】
                      ▼
                  向量数据库
                      │ 【用户提问 → 向量化】
                      ▼
       【相似度算法检索】Top-K 相关 Chunk
                      │ 【Rerank 重排序】
                      ▼
                 【LLM 生成答案】
```

### 核心速记口诀（全文复习专用）

- **Chunking 切块先做好，语义完整不能少**
- **重叠 10~20%，边界知识不丢跑**
- **Embedding 向量化，Encoder 模型干这事**
- **语义相近向量近，计算机比对数字不读字**
- **余弦相似度首选，点积归一化提速**
- **私域知识靠 RAG，准确可溯不微调**
